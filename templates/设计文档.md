## 一、背景与目标

### 1.1 问题背景

现有**单体应用**随业务高速增长已遇到严重性能与运维瓶颈，核心痛点如下：

| 指标 | 现状 | 问题表现 |
|:---|:---|:---|
| 日订单量 | 10万 → **80万**（持续增长） | 系统负载逼近上限 |
| 高峰期 QPS | **5000+** | 数据库连接池频繁打满 |
| 单次部署耗时 | **30 分钟** | 需全量停服，影响业务连续性 |
| 系统可用性 | **99.5%** | 年累计不可用时间约 **43.8 小时** |

> ⚠️ **核心矛盾：** 单体架构下，所有模块共享数据库连接池与部署生命周期，任意模块的性能退化或发布变更都将影响全局服务可用性。

### 1.2 设计目标

| # | 目标 | 现状 | 目标值 | 提升幅度 |
|:---:|:---|:---:|:---:|:---:|
| 1 | 系统可用性 | 99.5% | **99.99%** | ~10x 减少停机 |
| 2 | 日订单处理能力 | 80万 | **200万** | 2.5x |
| 3 | 单次部署耗时 | 30 分钟 | **≤ 5 分钟** | 6x |
| 4 | 弹性伸缩能力 | 整体扩容 | **各服务独立扩缩容** | — |

## 三、详细设计

### 3.1 订单状态机

订单生命周期通过有限状态机进行管理，状态码定义与流转规则如下：

| 状态码 | 状态名 | 可流转至 |
|:---:|:---|:---|
| `0` | 待支付 | `1`（已支付）、`4`（已取消） |
| `1` | 已支付 | `2`（已发货）、`5`（已退款） |
| `2` | 已发货 | `3`（已完成） |
| `3` | 已完成 | — （终态） |
| `4` | 已取消 | — （终态） |
| `5` | 已退款 | — （终态） |

```
              ┌──────────┐    支付成功    ┌──────────┐   商家发货   ┌──────────┐   确认收货   ┌──────────┐
              │  0-待支付  │─────────────►│  1-已支付  │───────────►│  2-已发货  │───────────►│  3-已完成  │
              └─────┬─────┘              └─────┬─────┘            └──────────┘            └──────────┘
                    │ 用户/超时取消              │ 退款申请
                    ▼                          ▼
              ┌──────────┐              ┌──────────┐
              │  4-已取消  │              │  5-已退款  │
              └──────────┘              └──────────┘
```

### 3.2 创建订单核心流程

创建订单涉及多服务协同，采用 **Saga 编排模式**保证最终一致性：

1. **网关层**接收请求，路由至 `order-service`
2. `order-service` 通过 gRPC 调用 `user-service` 校验用户信息与地址
3. `order-service` 通过 gRPC 调用 `inventory-service` 执行**库存预占**
4. `order-service` 写入订单记录（状态 = `0-待支付`）
5. `order-service` 发布 `order.created` 消息至 RabbitMQ
6. `notification-service` 消费消息，发送下单通知

> 💡 **补偿机制：** 若步骤 4 写入失败，系统自动调用 `inventory-service` 释放已预占库存（Saga 补偿事务）。

### 3.3 消息队列设计

使用 RabbitMQ **Topic Exchange** 实现事件驱动架构：

- **Exchange 名称：** `order.topic`
- **Exchange 类型：** `TopicExchange`

| Routing Key | 触发时机 | 消费方 | 处理动作 |
|:---|:---|:---|:---|
| `order.created` | 订单创建成功 | `inventory-service` | 库存扣减确认 |
| `order.paid` | 支付成功回调 | `order-service`、物流服务 | 通知发货 |
| `order.cancelled` | 订单取消 | `inventory-service` | 库存释放 |
| `order.completed` | 订单完成 | 积分服务、`notification-service` | 积分奖励、完成通知 |

消息体示例：

```json
{
  "event_id": "evt_20240101_abc123",
  "event_type": "order.created",
  "timestamp": "2024-01-01T12:00:00Z",
  "payload": {
    "order_id": 100001,
    "order_no": "ORD20240101120000001",
    "user_id": 88001,
    "items": [
      { "sku_id": "SKU001", "quantity": 2 },
      { "sku_id": "SKU002", "quantity": 1 }
    ],
    "total_amount": 299.00
  }
}
```

### 3.4 服务间通信策略

| 场景 | 协议 | 说明 |
|:---|:---:|:---|
| 服务内部同步调用 | **gRPC** | 高性能、强类型、适合低延迟内部通信 |
| 对外暴露接口 | **REST** | 通过 Kong 网关统一暴露，兼容性好 |
| 异步事件通知 | **AMQP** | 通过 RabbitMQ 解耦，支持削峰填谷 |

## 四、数据库 / 数据模型

### 4.1 订单主表（`orders`）

```sql
CREATE TABLE orders (
    id               BIGINT          PRIMARY KEY AUTO_INCREMENT,
    order_no         VARCHAR(32)     NOT NULL UNIQUE             COMMENT '订单编号',
    user_id          BIGINT          NOT NULL                    COMMENT '用户ID',
    total_amount     DECIMAL(10,2)   NOT NULL                    COMMENT '订单总金额',
    discount_amount  DECIMAL(10,2)   DEFAULT 0                   COMMENT '优惠金额',
    pay_amount       DECIMAL(10,2)   NOT NULL                    COMMENT '实付金额',
    status           TINYINT         NOT NULL DEFAULT 0          COMMENT '0-待支付 1-已支付 2-已发货 3-已完成 4-已取消 5-已退款',
    payment_method   VARCHAR(20)                                 COMMENT '支付方式',
    address_snapshot JSON                                        COMMENT '地址快照',
    remark           VARCHAR(500)                                COMMENT '备注',
    created_at       DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at       DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,

    INDEX idx_user_id         (user_id),
    INDEX idx_order_no        (order_no),
    INDEX idx_status_created  (status, created_at)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 4.2 字段说明

| 字段 | 类型 | 必填 | 默认值 | 说明 |
|:---|:---|:---:|:---:|:---|
| `id` | `BIGINT` | ✅ | AUTO_INCREMENT | 主键 |
| `order_no` | `VARCHAR(32)` | ✅ | — | 业务订单编号，全局唯一 |
| `user_id` | `BIGINT` | ✅ | — | 关联用户 ID |
| `total_amount` | `DECIMAL(10,2)` | ✅ | — | 订单总金额（优惠前） |
| `discount_amount` | `DECIMAL(10,2)` | ❌ | `0` | 优惠减免金额 |
| `pay_amount` | `DECIMAL(10,2)` | ✅ | — | 实际支付金额 |
| `status` | `TINYINT` | ✅ | `0` | 订单状态，见状态机定义 |
| `payment_method` | `VARCHAR(20)` | ❌ | `NULL` | 支付方式（如 `alipay`、`wechat`） |
| `address_snapshot` | `JSON` | ❌ | `NULL` | 下单时收货地址快照（防止地址修改影响历史订单） |
| `remark` | `VARCHAR(500)` | ❌ | `NULL` | 用户备注 |
| `created_at` | `DATETIME` | ✅ | `CURRENT_TIMESTAMP` | 创建时间 |
| `updated_at` | `DATETIME` | ✅ | `CURRENT_TIMESTAMP` | 最后更新时间 |

### 4.3 索引设计说明

| 索引名 | 索引字段 | 用途 |
|:---|:---|:---|
| `idx_user_id` | `user_id` | 用户维度订单查询 |
| `idx_order_no` | `order_no` | 订单号精确查询 |
| `idx_status_created` | `status, created_at` | 按状态 + 时间范围的组合查询与分页 |

#### 5.2.1 创建订单

```

Content-Type: application/json
Authorization: Bearer <token>
```

## 六、非功能性需求

### 6.1 性能要求

| 指标 | 目标值 | 测量方式 |
|:---|:---|:---|
| 接口响应时间（P99） | **< 200ms** | Prometheus histogram |
| 系统吞吐量 | **> 10,000 QPS** | 网关层统计 |
| 日订单处理能力 | **200 万单 / 天** | 业务指标监控 |
| 系统可用性 | **99.99%** | 年停机时间 ≤ 52.6 分钟 |

### 6.2 数据一致性

> 🔑 **关键决策：** 采用 **Saga 模式**（编排式）处理分布式事务，放弃强一致性，拥抱**最终一致性**。

- 每个服务维护本地事务
- 通过事件消息触发下游服务
- 每个正向操作配对一个**补偿操作**
- 引入 Saga 执行协调器统一编排

### 6.3 可观测性

| 层面 | 工具 | 职责 |
|:---|:---|:---|
| **指标监控** | Prometheus + Grafana | QPS、延迟、错误率、资源使用率 |
| **链路追踪** | Jaeger | 跨服务调用链路可视化、瓶颈定位 |
| **日志管理** | ELK Stack | 集中式日志收集、检索与告警 |

### 6.4 安全性

- 所有外部接口通过 **Kong 网关**统一鉴权
- 服务间通信使用 **mTLS** 加密
- 敏感数据（手机号、地址等）落库**脱敏**处理

### 6.5 可扩展性

- 各微服务**独立部署**、独立扩缩容
- 无状态设计，支持**水平扩展**
- 库存服务基于 Redis，天然支持**高并发读写**
