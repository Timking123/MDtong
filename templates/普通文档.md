# Python 项目开发规范与最佳实践指南

> 📌 本指南面向团队新成员，涵盖环境搭建、代码规范、项目结构、测试策略及常用工具等内容，帮助你快速融入团队开发流程。

## 一、环境搭建

### 1.1 Python 版本管理

推荐使用 **pyenv** 管理 Python 版本。安装命令如下：

```bash
curl https://pyenv.run | bash
```

安装完成后，在 `~/.bashrc`（或 `~/.zshrc`）中添加以下配置：

```bash
export PATH="$HOME/.pyenv/bin:$PATH"
eval "$(pyenv init -)"
```

> ⚠️ 项目统一使用 **Python 3.11+**，请确保本地版本符合要求。

### 1.2 虚拟环境

使用标准库 `venv` 创建项目虚拟环境：

```bash
python -m venv .venv
source .venv/bin/activate  # Linux/macOS
```

### 1.3 依赖管理

采用 **pip + requirements.txt** 方案管理依赖，使用锁文件固定版本：

```bash
pip install -r requirements.txt       # 安装依赖
pip freeze > requirements.lock         # 锁定版本
```

### 1.4 IDE 配置

推荐使用 **VS Code**，以下为必装插件：

| 插件名称           | 用途           |
| :----------------- | :------------- |
| **Python**         | Python 语言支持 |
| **Pylint**         | 代码静态检查   |
| **Black Formatter** | 代码自动格式化 |
| **GitLens**        | Git 增强工具   |

## 二、代码规范

### 2.1 命名规范

| 类型       | 风格                   | 示例                  |
| :--------- | :--------------------- | :-------------------- |
| 变量 / 函数 | `snake_case`           | `process_data`        |
| 类         | `PascalCase`           | `DataProcessor`       |
| 常量       | `UPPER_SNAKE_CASE`     | `MAX_RETRY_COUNT`     |
| 私有属性   | 单下划线前缀           | `_internal_cache`     |

### 2.2 类型注解

类型注解是**必须的**。函数签名示例：

```python
def process_data(
    items: list[dict[str, Any]],
    threshold: float = 0.5
) -> tuple[int, list[str]]:
    ...
```

### 2.3 文档字符串

统一采用 **Google 风格** 的文档字符串：

```python
def calculate_score(raw_data: list[float], weights: dict[str, float]) -> float:
    """计算加权评分。

    Args:
        raw_data: 原始数据列表，每个元素为 float。
        weights: 权重字典，键为类别名，值为权重系数。

    Returns:
        归一化后的评分，范围 0-100。

    Raises:
        ValueError: 当 raw_data 为空时抛出。
    """
    pass
```

## 三、项目结构

标准项目目录结构如下：

```
project-root/
├── src/
│   └── package_name/        # 主代码包
│       ├── __init__.py
│       └── ...
├── tests/                   # 测试代码
│       ├── __init__.py
│       └── ...
├── docs/                    # 项目文档
├── scripts/                 # 工具脚本
├── pyproject.toml           # 项目配置（构建、工具链）
├── requirements.txt         # 项目依赖
├── README.md                # 项目说明
└── .gitignore               # Git 忽略规则
```

> 💡 `pyproject.toml` 是现代 Python 项目的**核心配置文件**，Black、Ruff 等工具的配置也统一放置于此。

## 四、测试

### 4.1 框架与运行

测试框架统一使用 **pytest**，运行命令：

```bash
pytest -v --cov=src --cov-report=html
```

### 4.2 命名约定

| 项目     | 规则                 | 示例                    |
| :------- | :------------------- | :---------------------- |
| 测试文件 | `test_模块名.py`     | `test_calculator.py`    |
| 测试函数 | `test_功能描述`      | `test_add_positive`     |

### 4.3 覆盖率要求

- **单元测试**覆盖率要求 ≥ **85%**
- **集成测试**需覆盖所有核心业务流程
- Mock 外部依赖使用 `unittest.mock` 或 `pytest-mock`

### 4.4 测试示例

```python
import pytest
from myapp.calculator import add, divide


def test_add_positive():
    assert add(2, 3) == 5


def test_divide_by_zero():
    with pytest.raises(ZeroDivisionError):
        divide(10, 0)
```

## 五、常用工具

### 5.1 代码格式化工具对比

| 工具         | 特点             | 优势           | 劣势           | GitHub Stars |
| :----------- | :--------------- | :------------- | :------------- | :----------: |
| **Black** ⭐  | 零配置，开箱即用 | 格式统一，速度快 | 不可自定义规则 |     35k      |
| autopep8     | 遵循 PEP 8       | 可自定义       | 格式不够统一   |      4k      |
| yapf         | Google 出品      | 高度可配置     | 速度慢，配置复杂 |     14k      |

> ✅ **团队推荐：Black**，行长度设置为 **120 字符**，配置写入 `pyproject.toml`：

```toml
[tool.black]
line-length = 120
target-version = ["py311"]
```

### 5.2 静态检查

推荐使用 **Ruff**，相比 Pylint 快 **10–100 倍**，配置同样放在 `pyproject.toml` 中：

```toml
[tool.ruff]
line-length = 120
select = ["E", "F", "W", "I"]
```

### 5.3 Pre-commit 钩子

通过 `.pre-commit-config.yaml` 配置提交前自动检查，确保代码在进入仓库前符合规范：

```yaml
repos:
  - repo: https://github.com/psf/black
    rev: 24.4.2
    hooks:
      - id: black
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.4.4
    hooks:
      - id: ruff
```

### 5.4 性能优化速查

| 场景               | 推荐做法                       | 预期提升   |
| :----------------- | :----------------------------- | :--------: |
| 大列表操作         | 用**列表推导式**替代 `for` 循环 |  ~30%     |
| 频繁字符串拼接     | 用 `"".join()` 替代 `+=`       |  ~50%     |
| 重复计算缓存       | 使用 `functools.lru_cache`     |  视场景而定 |
| 并发 I/O           | 使用 `asyncio` 或 `concurrent.futures` | 显著 |
