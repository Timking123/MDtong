## 一、需求背景

### 1.1 业务背景

当前 App 的推送功能存在**架构分散、体验割裂**的问题，促销推送、订单通知、系统消息各走独立通道，缺乏统一的消息管理中心。这导致了以下核心矛盾：

| 维度 | 现状 | 问题影响 |
|:---:|:---|:---|
| **通道管理** | 各业务线独立对接推送通道 | 消息送达率仅 **75%**，重要通知易丢失 |
| **用户体验** | 无统一消息入口，无偏好设置 | 用户频繁投诉被营销推送打扰 |
| **运营效率** | 无模板管理，推送策略粗放 | 推送内容质量参差不齐，缺少数据闭环 |

### 1.2 用户痛点

- 重要通知（如订单发货、退款到账）**送达不及时或丢失**
- 营销推送过于频繁，用户**无法自主控制接收偏好**
- 历史消息**无处可查**，错过即丢失

### 1.3 市场分析

> 竞品分析发现，头部 App（淘宝、京东、美团等）均已建设**统一消息中心**，提供分类查看、偏好设置、免打扰等标准化功能，已成为行业基础能力。

**客户满意度调研数据**：

| 调研指标 | 当前数据 | 行业基准 |
|:---:|:---:|:---:|
| 通知体验评分 | **3.2** / 5 分 | 4.0 / 5 分 |
| 消息送达率 | 75% | 95%+ |
| 推送相关投诉占比 | 较高 | 低 |

## 二、需求目标

### 2.1 核心目标

建设**统一的消息推送中心**，整合全业务线推送通道，提供用户侧消息管理能力与运营侧推送配置工具，全面提升用户体验和消息送达率。

### 2.2 关键指标（KPI）

| KPI 指标 | 当前值 | 目标值 | 提升幅度 |
|:---|:---:|:---:|:---:|
| 消息送达率 | 75% | **95%** | +20pp |
| 通知设置页访问后投诉率 | 基准值 | 下降 **50%** | -50% |
| 消息中心日活渗透率 | — | **30%** | 新增指标 |

### P0 — 必须实现

#### F01 统一消息中心页面

- **功能描述**：在 App 首页提供统一消息入口，展示未读消息数角标。进入消息中心后，所有消息按时间倒序排列，支持按类型分 Tab 浏览。
- **消息分类 Tab**：

| Tab 名称 | 消息类型 | 典型内容示例 |
|:---:|:---:|:---|
| 交易消息 | 订单状态变更 | 订单发货、退款到账、签收提醒 |
| 活动消息 | 营销推广 | 优惠券到账、限时秒杀、会员活动 |
| 系统通知 | 平台公告 | 版本更新、隐私政策变更、维护通知 |

- **用户故事**：

> 作为一名用户，我希望在一个统一的地方查看所有消息，这样就不会遗漏任何重要通知。

- **验收标准**：
  - [x] 消息按时间**倒序**展示
  - [x] 支持**下拉刷新**和**上滑加载更多**（分页加载，每页 20 条）
  - [x] 未读消息有明确的**视觉标记**（如加粗/圆点）
  - [x] 支持**"全部已读"** 一键操作
  - [x] 首页角标实时展示未读消息总数

#### F02 推送偏好设置

- **功能描述**：用户可按消息类型独立控制推送通知的开关，并可设置免打扰时段。

| 消息类型 | 默认状态 | 用户可关闭 | 说明 |
|:---:|:---:|:---:|:---|
| 交易消息 | ✅ 开启 | ❌ 不可 | 涉及资金和订单安全，**强制推送** |
| 活动消息 | ✅ 开启 | ✅ 可以 | 尊重用户偏好 |
| 系统通知 | ✅ 开启 | ✅ 可以 | 尊重用户偏好 |

- **用户故事**：

> 作为一名用户，我希望能自主控制接收哪些类型的推送通知，并设置夜间免打扰，这样既不会错过重要消息，也不会被无关推送打扰。

- **验收标准**：
  - [x] 偏好设置修改后**实时生效**
  - [x] 服务端持久化存储用户偏好配置
  - [x] 免打扰时段内，推送**静默不弹通知**，但消息仍写入消息中心，可正常查看
  - [x] 交易消息开关置灰且不可操作，附注文字说明

#### F03 消息模板管理后台

- **功能描述**：为运营人员提供消息模板的创建、编辑、预览和管理功能，支持变量动态替换。

- **模板结构**：

| 字段 | 是否必填 | 说明 |
|:---:|:---:|:---|
| 模板标题 | ✅ | 推送通知标题 |
| 模板正文 | ✅ | 推送内容，支持变量插入 |
| 配图 | ❌ | 可选图片 URL |
| 跳转链接 | ❌ | 用户点击后跳转的 DeepLink / URL |

- **支持的模板变量示例**：

```text
{用户名}     → 张三
{订单号}     → 2026041500001
{商品名}     → iPhone 18 Pro
{金额}       → ¥8,999.00
{时间}       → 2026-04-15 14:30
```

- **用户故事**：

> 作为运营人员，我希望通过模板化的方式快速创建推送消息，并能预览最终效果，这样可以保障推送内容的一致性和专业性。

- **验收标准**：
  - [x] 模板支持 **标题 + 正文 + 图片 + 跳转链接** 四要素
  - [x] 变量插入时系统**自动校验**变量名合法性（非法变量名红色提示）
  - [x] 提供**实时预览**功能，模拟终端展示效果
  - [x] 模板支持**保存草稿**和**发布**两种状态

### P1 — 应该实现

#### F04 智能推送策略

- **功能描述**：系统基于用户历史活跃时段数据，自动选择最优推送时机。同时支持 A/B 测试，对比不同推送文案的点击效果。
- **用户故事**：

> 作为运营人员，我希望系统能智能选择用户最可能打开推送的时间窗口进行推送，并通过 A/B 测试找到最优文案，以提升推送点击率。

- **验收标准**：
  - [ ] 推送时机优化后，整体点击率提升 **≥15%**
  - [ ] A/B 测试支持至少 **2 组**对照实验
  - [ ] 实验结果提供**置信度**和**统计显著性**指标

#### F05 消息已读回执

- **功能描述**：完整追踪消息生命周期的各关键节点状态。

| 追踪节点 | 触发时机 | 说明 |
|:---:|:---|:---|
| 已发送 | 推送请求成功提交至推送通道 | 服务端记录 |
| 已送达 | 设备成功接收推送 | 推送 SDK 回调 |
| 已展示 | 推送通知在通知栏展示 | 客户端埋点 |
| 已点击 | 用户点击推送通知 | 客户端埋点 |

- **用户故事**：

> 作为运营人员，我希望能看到每条推送的送达、展示、点击全链路数据，以便分析推送效果并持续优化。

- **验收标准**：
  - [ ] 各节点埋点数据**准确率 ≥ 99%**
  - [ ] 数据面板支持**实时**和**T+1**两种查看模式

#### F06 富媒体消息

- **功能描述**：推送消息支持图文混排和交互式按钮操作，用户可在推送通知中直接完成简单交互（如一键评价、确认收货）。
- **用户故事**：

> 作为用户，我希望在推送消息中直接完成评价等操作，而不需要层层跳转。

- **验收标准**：
  - [ ] 支持**图文混排**消息格式
  - [ ] 支持至少 **2 个**可交互按钮
  - [ ] iOS / Android 双平台样式**一致性**达标

## 五、非功能需求

### 5.1 性能要求

| 指标 | 要求 |
|:---|:---|
| 消息中心页面加载时间 | 首屏 **≤ 1.5 秒**（4G 网络） |
| 推送下发延迟 | 事件触发到推送送达 **≤ 5 秒**（实时类消息） |
| 单次推送用户上限 | **100 万**，超出需分批处理 |
| 消息列表分页 | 每页 20 条，接口响应 **≤ 500ms** |

### 5.2 安全要求

- 推送通道使用 **HTTPS / TLS** 加密传输
- 用户偏好等敏感操作需要**登录态校验**
- 后台模板管理需 **RBAC 权限控制**，区分运营编辑和审核角色
- 推送内容需通过**敏感词过滤**机制

### 5.3 兼容性要求

| 平台 | 推送通道 | 最低支持版本 |
|:---:|:---:|:---:|
| iOS | `APNs` | iOS 13+ |
| Android | `FCM` + 厂商通道 | Android 8.0+ |

### 5.4 频率限制

> ⚠️ **重要限制**：单用户每天最多接收 **10 条营销推送**（活动消息类型），交易消息和系统通知不受此限制。

### 5.5 数据存储策略

| 策略 | 说明 |
|:---|:---|
| 热存储 | 最近 **90 天**消息，支持实时查询 |
| 冷归档 | 超过 90 天自动归档至冷存储，不再对用户展示 |

## 六、交互流程

### 6.1 核心流程 — 接收推送并查看详情

```
┌──────────────────────────────────────────────────────────────┐
│                        推送核心流程                           │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  ① 服务端触发推送事件（如订单发货）                            │
│         │                                                    │
│         ▼                                                    │
│  ② 推送引擎检查用户偏好 & 免打扰设置                          │
│         │                                                    │
│         ├── 偏好关闭 ──▶ 仅写入消息中心，不推送               │
│         │                                                    │
│         ├── 免打扰中 ──▶ 消息入库，延迟至时段结束后推送        │
│         │                                                    │
│         ▼                                                    │
│  ③ 检测设备类型，调用对应推送通道                              │
│         │                                                    │
│         ├── iOS   ──▶ APNs                                   │
│         ├── Android ─▶ FCM / 厂商通道                        │
│         │                                                    │
│         ▼                                                    │
│  ④ 用户设备展示推送通知（标题 + 摘要）                         │
│         │                                                    │
│         ▼                                                    │
│  ⑤ 用户点击推送 ──▶ App 打开对应详情页                        │
│         │                                                    │
│         ▼                                                    │
│  ⑥ 标记消息已读，上报点击事件埋点                              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 6.2 流程步骤详解

1. **服务端触发推送事件**：业务系统（如订单服务）通过统一推送 API 发起推送请求，传入用户 ID、消息类型、模板 ID 及变量参数。
2. **推送引擎检查用户偏好和免打扰设置**：查询 `用户推送偏好表`，判断该消息类型推送开关是否开启、当前时间是否在免打扰时段内。
3. **调用对应推送通道**：根据用户设备类型选择 `APNs`（iOS）或 `FCM` / 厂商通道（Android），通过极光推送 SDK 统一下发。
4. **设备展示推送通知**：用户终端通知栏展示推送，包含标题与摘要内容。
5. **用户点击推送通知**：App 解析 DeepLink，跳转至对应详情页面（如订单详情、活动页）。
6. **标记已读 & 上报埋点**：客户端标记该消息为已读，更新消息中心角标数，同时上报 `click` 事件至数据平台。

push.platform = jpush.all_
